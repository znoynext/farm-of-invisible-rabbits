# Farm of Invisible Rabbits

Интерактивная система наблюдения за невидимыми кроликами по косвенным сигналам на ферме. Приложение помогает оценить возможное количество животных, показывает качество доказательной базы и объясняет, какие наблюдения сформировали результат.

## О проекте

Самих кроликов увидеть нельзя, но можно анализировать оставленные ими следы: пропавшую морковь, новые ямки, срабатывания датчика движения и шорох. Farm of Invisible Rabbits объединяет эти наблюдения в прозрачную воспроизводимую оценку и позволяет проверить, как изменится вывод при других исходных данных.

Это статическое SPA с полностью русским пользовательским интерфейсом. Расчёты выполняются локально и не зависят от runtime AI/LLM.

## Выбранный сценарий

**Scenario 2 — invisible rabbit farm.**

Интерфейс построен вокруг наблюдений, объяснимости и безопасных экспериментов: пользователь сначала видит основной ответ, затем изучает пространственное распределение следов, причины оценки, What-if-сценарий и рекомендации.

## Возможности

- основной estimate предполагаемого количества кроликов, уровень активности и confidence;
- динамически рассчитанный dominant evidence — главный источник активности;
- интерактивная Farm Map с агрегацией наблюдений по зонам;
- Evidence-раздел с вкладом типов сигналов и формулой каждого события;
- What-if Scenario Lab с предварительным просмотром результата до применения;
- детерминированные рекомендации с объяснением причин;
- добавление, редактирование, удаление и восстановление наблюдений;
- настройка sensitivity и весов четырёх типов сигналов;
- локальное сохранение наблюдений, настроек модели и intro preference;
- отдельный раздел «Работа с ИИ» с публичным AI Worklog;
- адаптивный интерфейс, клавиатурное управление и поддержка `prefers-reduced-motion`.

## UX-концепция

Основной путь выстроен как последовательное повествование:

**Ответ → Пространство → Почему → Эксперимент → Действие → Управление данными и моделью.**

Hero остаётся главным визуальным и смысловым центром. Карта, Evidence, What-if и Recommendations раскрывают один и тот же результат с разных сторон, а Signals и Model работают как отдельные management-слои. Поэтому интерфейс не строится вокруг сетки KPI-карточек и не имитирует типовой административный dashboard.

## Как работает модель

Каждое наблюдение содержит тип сигнала, локацию, количество, интенсивность и время. После пяти однотипных следов действует diminishing returns: дополнительные повторы продолжают усиливать evidence, но медленнее.

```text
effectiveCount = count, если count <= 5
effectiveCount = 5 + sqrt(count - 5), если count > 5

eventImpact = effectiveCount × signalWeight × (intensity / 10)
estimatedRabbits = max(0, round(sum(eventImpact) × sensitivity))
```

`count` задаёт объём наблюдения, `intensity` — его силу, `signalWeight` — относительную значимость типа следа, а `sensitivity` масштабирует итоговую оценку. Все вычисления детерминированы; округление для интерфейса не меняет внутреннюю точность contributions.

### Как считается уверенность

Confidence объединяет четыре характеристики доказательной базы:

- разнообразие типов сигналов;
- покрытие разных локаций;
- количество наблюдений;
- среднюю интенсивность.

Это эвристическая оценка качества и разнообразия наблюдений, а не статистическая вероятность того, что точное количество кроликов определено правильно. Confidence всегда находится в диапазоне 0–100 и не зависит от sensitivity или весов сигналов.

Подробная каноническая спецификация находится в [`docs/domain-model.md`](docs/domain-model.md).

## Интерактивная карта

Farm Map агрегирует события по локациям и показывает уровень активности каждой зоны. Выбор зоны связывает карту с соответствующими Evidence-элементами; выбор типа evidence подсвечивает все относящиеся к нему локации. Connected state появляется только после явного действия пользователя.

Для неизвестных локаций используется детерминированное размещение в отдельной fallback-области, поэтому одинаковые данные всегда дают одинаковую карту.

## What-if Scenario Lab

Лаборатория работает с временной копией observations. Изменение интенсивности сразу пересчитывает preview для Hero, Farm Map, Evidence и Recommendations, но не меняет сохранённые данные до действия «Применить к данным».

После применения обновляются только observations. Sensitivity и веса модели What-if-экспериментом не изменяются; сброс сценария возвращает интерфейс к базовым данным.

## Управление наблюдениями

Раздел «Сигналы» поддерживает добавление, редактирование, удаление одного события, удаление всех событий и восстановление стартового набора. Входные данные проверяются общей Zod-схемой.

Восстановление исходных данных заменяет пользовательские observations, поэтому требует явного подтверждения. Отмена и Escape не меняют данные.

## Модель оценки

Пользователь может менять:

- sensitivity в диапазоне `0.5–1.5`;
- веса типов сигналов в диапазоне `0–3`;
- стандартные настройки можно восстановить отдельным действием.

Изменения сразу пересчитывают estimate, dominant evidence, impacts локаций, карту, Evidence и Recommendations. Confidence при этом остаётся неизменным. Настройки сохраняются локально и не сбрасывают observations или intro preference.

## Архитектура

- `src/domain` — чистые TypeScript-функции расчётов, агрегации, рекомендаций и validation;
- `src/app/state` — React Context + `useReducer`, versioned persistence и derived analytics;
- `src/features` — пользовательские разделы, получающие готовые domain results;
- `src/components` — переиспользуемые UI primitives и signal marks;
- `src/data` — стартовые observations и публичная выборка AI Worklog;
- `e2e` — сквозные пользовательские сценарии Playwright.

Persistent state содержит `schemaVersion`, observations и model settings. Intro preference хранится отдельно. Scenario preview, диалоги и связанный выбор Map/Evidence не сохраняются. Загруженные данные валидируются через Zod; повреждённое или несовместимое состояние безопасно заменяется defaults.

В проекте нет backend, database, authentication и runtime LLM API.

## Запуск локально

Для воспроизводимого запуска используйте Node.js 24 и npm.

```bash
git clone https://github.com/znoynext/farm-of-invisible-rabbits.git
cd farm-of-invisible-rabbits
npm ci
npm run dev
```

Vite выведет актуальный локальный URL в терминале.

## Production build

```bash
npm run build
npm run preview
```

`build` выполняет строгую TypeScript-проверку и собирает статическое приложение, а `preview` запускает локальный сервер для проверки production-сборки.

## Тестирование

```bash
npm run lint
npm run test:run
npm run test:e2e
npm run build
```

Проверки покрывают domain invariants, React UI, persistence fallbacks, интеграцию Map/Evidence/What-if, accessibility-контракты и сквозные пользовательские сценарии. Финальный pre-release gate прошёл lint, unit/UI, E2E и production build; изменяющиеся численные счётчики тестов намеренно не закреплены в README.

## Как использовался ИИ

Разработка велась как управляемый AI-first процесс:

**Codex / AI coding assistant → repository rules → специализированные project skills → поэтапные implementation prompts → автоматические проверки → независимые AI-assisted audits → решение человека → точечная remediation → повторный gate.**

Человек задавал product direction и scope, принимал решения по audit findings, корректировал или отклонял предложения ИИ и контролировал обязательные проверки перед публикацией. Внешние AI-assisted audits не выдаются за human review. Расчётная модель, продуктовые границы и фактические результаты проверок фиксировались в repository sources of truth.

## AI Worklog

В интерфейсе есть отдельный раздел «Работа с ИИ». Он содержит семь выбранных публичных checkpoints из более полного рабочего журнала. Каждый checkpoint показывает:

- задачу и краткое содержание prompt;
- предложение ИИ;
- решение человека;
- фактические изменения;
- выполненную validation.

Полные рабочие notes находятся в [`docs/ai-worklog-notes.md`](docs/ai-worklog-notes.md). В публичную UI-выборку не включаются chain-of-thought, секреты, credentials или локальные персональные данные.

## Ограничения

- модель является прозрачной демонстрационной эвристикой;
- trained ML model и научная валидация отсутствуют;
- результат не является реальной системой обнаружения животных;
- данные и настройки хранятся только в `localStorage` текущего браузера;
- синхронизации между устройствами и пользовательских аккаунтов нет;
- runtime AI/LLM не используется для вычислений или рекомендаций.
